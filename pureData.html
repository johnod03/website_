Pure data truely is a phenomenal software package. It's hard to believe that such a powerful package is completely free. A true testament to the open source movement. For this workshop component, we were encouraged to experiement with PD and see what we could come up with. The possibilities are endless. Although PD is originally designed to route and control audio and number data, it is possible to take it as far as the imagination will allow, especially with the use of GEM, the graphics side of PD. This link contains a zipped folder of all PD code/patches used in my projects. Note, these are still works in progress and should provide some explaination as to how I used PD....

Frame Differencing

An experiment with frame differencing. I wanted to explore this technique as it is something I could use in my dissertation project when it comes to sonifying video data. The original movie is converted from colour to greyscale in order to make the analysis less complex.

Noise Drone

I have found myself extremely interested in the creation of drones using computer music techniques. Again, These could play a part in my dissertation project so I am trying to use Mikael's workshop components to explore methods that will inform the final project somehow.

noiseDrone from John O' Dwyer on Vimeo.

This noise drone patch uses a number of oscillators that are modulated over and over to create the drone sounds. A noise generator is used to add extra interest to the texture.

Synth Demo

A demo of my drone synth in action. Here, I used Pduino combined with Firmata to allow interactivity with the patch. A flex sensor and a potentiometer are connected to the Arduino and they are mapped using a one-to-many stratagey. Multiple parameters are affected by a single controller.

synthDemoSound from John O' Dwyer on Vimeo.

In the video above I tried to capture the interactions occuring and demonstrate the resulting parameter changes in PD.

GEM Demo

Screeen capture of GEM patch running alongside the synth patch with common mappings.

GEM with Sound from John O' Dwyer on Vimeo.

The GEM patch allows the creation and control of graphics. Control is again provided by interacting with the hardware sensors and Arduino. The pot moves the shapes in opposite directions and the flex is controlling the colour of the graphic elements.

Flex Sensor/Arduino Circuit

Close up of Arduino and flex sensor wiring. The flex is mapped using one-to-many mappings.

As well as controlling the roomsize parameter of the freeverb~ object, the flex sensor also controls the RGB values of both the geometric shapes and the particle system. This provides an interesting multi-modal interactive application where audio and video components are affected by a single hardware controller.

Below are some stills and screenshots of coding examples and renderings from the GEM window. Some details are given as to how the PD patches work and the screenshots help to illustrate the intentions.

Potentiometer

A close up of the potentiometer. The pot was mounted upon a cigar box for another project and this was used here once again.

The values from the potentiometer range from 0 - 1023 when they are sent to the serial port. They are then scaled using PD's autoscale object to 0 - 100. Once they are available in PD, it is possible to scale them to any range.

GEM Patch

The GEM patch draws a circle, a square and a particl system to the GEM window. Each of theses elements have a number of controllable parameters. In this case, the colour, scale and X position of the square and circle are mapped to the pot.

In the screenshot it is possible to see the mappings and the scale object taking place.

Variable delay

The patch snippet in the image below shows a system for automating the delay time in a variable delay system.

The values of the delay time are set to change according to the metro time (2 - 2.5 seconds , left - right channels respectively) and a random delay time between 100 - 1100 is generated and sent to the variable delay.

Freeverb~

The freeverb~ effect has controllable parameters - roomsize, wet and dry. These affect the 'size' and 'space' of the freeverb~ object.

The roomsize is mapped and scaled (0 - 1) to the value from the flex sensor. It would have been possible to map the wet and dry mix but I figured this would be etter achieved with other sensors.

“Some people think electronic music is cold, but I think that has more to do with the people listening than the actual music itself.” ― Imogen Heap.

FM Patch

The FM synth comprises, mainly, of 2 frequency modulated oscillators and a sawtooth wave generator.

The central frequency or carrier of each osc~ is offset, initially, by 1 Hz. This is in order to create a ring modulator affect. The phasor~ adds a sawtooth wave into the system. Each of the oscillators and the phasor~ are sent through a bandpass~ filter which is controlled by the Arduino sensors also.
